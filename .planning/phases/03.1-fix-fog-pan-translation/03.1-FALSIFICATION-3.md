### Sub-section C — NEW failure surfaced (rotation-correlated fog mis-coverage)

#### Developer's verbatim observation

*"so the shader going outside of the screen happens when you zoom and rotate (which always happend when zooming with human finger) I think we should disable rotation for now, only pan and zoom, north is up. simplier"*

#### Interpretation against codebase

Plan 03.1-08's FOG-13 fix is `canvas.translate(-canvasOffset.dx, -canvasOffset.dy)` where `canvasOffset = Offset(matrix[12], matrix[13])` — i.e., it compensates ONLY the translation component of the canvas transform matrix. It does NOT compensate the rotation component (matrix[0], matrix[1], matrix[4], matrix[5]) or the scale component.

When MobileLayerTransformer applies a rotation transform to the painter's canvas (which happens during pinch-zoom-rotate gestures — two-finger pinch on iOS rarely keeps the angle at exactly 0, so any pinch-zoom triggers some rotation), the fog rect drawn at `Offset.zero & size` rotates WITH the canvas. The clip-path holes are in the original (untransformed) coordinate system per Plan 03.1-05's `computeFogClipPath(canvasOffset:)` shift, but the fog rect — being a `drawRect(Offset.zero & size, fogPaint)` — rotates with the prevailing canvas rotation. Result: the fog rect's bounding box no longer aligns with the viewport rect; wedges of un-fogged map appear at viewport corners (the corners of the rotated rect that are now outside the viewport stay un-painted; the corners of the viewport that the rotated rect doesn't cover stay un-fogged).

This was NOT surfaced in Walk #2 because:
1. The user's gestures may have happened to keep rotation near zero during the periods of high canvas-translation magnitude.
2. The fog-rect viewport-coverage gap (Walk #2 sub-section C — top-right + bottom-right diagonal strips) MAY have masked the rotation gap (the strips were aligned with the +canvasOffset translation, not with rotation). Now that Plan 03.1-08 closed the translation gap, the rotation gap becomes visible as a separate failure mode.

#### Developer's proposed mitigation

*"I think we should disable rotation for now, only pan and zoom, north is up. simplier"*

This is a **sensible scope reduction** — defers the canvas-rotation compensation problem until post-POC. FlutterMap supports `InteractionOptions(flags: InteractiveFlag.all & ~InteractiveFlag.rotate)` which disables rotation gestures while keeping pan + pinch-zoom. The MapCompass widget remains useful for displaying north (always 0° if rotation is disabled — the widget can be kept or removed as a separate UX-02 decision).
