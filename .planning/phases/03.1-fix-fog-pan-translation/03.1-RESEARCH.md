# Phase 3.1: Fix Fog Pan-Translation - Research

**Researched:** 2026-05-02
**Domain:** `flutter_map` 7.0.2 custom-layer translation propagation · `dart:ui` `Canvas.getTransform()` introspection · `flutter_test` 3.41 Canvas-mock pattern · per-paint JSONL diagnostic instrumentation
**Confidence:** **HIGH** — root cause confirmed by direct source-read of `flutter_map` 7.0.2 (`mobile_layer_transformer.dart` lines 1–27) AND of the in-tree shader (`assets/shaders/atmospheric_fog.frag` line 65 + 258) AND of the painter call site (`lib/presentation/widgets/fog_layer.dart` line 387). All three diagnostic possibilities discriminated by source-evidence, not inference.

## Summary

Source-read of `~/AppData/Local/Pub/Cache/hosted/pub.dev/flutter_map-7.0.2/lib/src/layer/shared/mobile_layer_transformer.dart` (lines 1–27) is decisive: **`MobileLayerTransformer` applies `Transform.rotate` ONLY — it does NOT translate the child Canvas at all.** flutter_map 7.0.2's translation propagation is per-layer-by-convention, not via a parent matrix transform. Each blessed layer (`TileLayer`, `CircleLayer`, `PolygonLayer`, `MarkerLayer`) carries its own per-paint `MapCamera.getOffsetFromOrigin(latLng)` / `latLngToScreenPoint(latLng)` projection that bakes in the camera's `pixelOrigin` (the panned world-origin) at every paint. The `TileLayer` does this via `Positioned(left: tileCoord * size - currentPixelOrigin.x, top: ...)` per tile widget; the `CircleLayer`'s painter calls `camera.getOffsetFromOrigin(circle.point)` per circle per paint.

The current `_FogPainter.paint()` does not consume `MapCamera.pixelOrigin`. It draws `canvas.drawRect(Offset.zero & size, Paint()..shader = liveShader)` filling the entire local widget. The clip path's holes ARE pan-aware (they project disc LatLngs through `camera.latLngToScreenPoint`), and the SDF cache rebuilds when the viewport bbox changes. **The remaining failure surface is the shader's `uOffset` uniform.** Reading `assets/shaders/atmospheric_fog.frag` line 65 and line 258:

```glsl
// World pan offset (in noise UV units). Lets the fog drift with the
// MapLibre camera so static fixtures don't appear to move when the user
// only pans. Slot 3..4.
uniform vec2  uOffset;
...
vec2 noiseUv = fragUv + uOffset;     // line 258
```

The atmospheric noise (FBM, curl-noise, all three parallax octaves, the boundary edge field, the hue field) is computed in `noiseUv = fragUv + uOffset`. The painter passes `offset: const (0.0, 0.0)` (`fog_layer.dart` line 387) — so `uOffset` is permanently `(0,0)` and the noise pattern is rigidly locked to screen-space. The visually dominant signal of the fog (the volumetric noise) does not translate with the camera regardless of how correctly the clip path or SDF behave.

This **is the root cause and it falls inside diagnostic possibility #1** (FogPainter paints in screen-space without consuming the camera translation transform), but with one important clarification vs the FALSIFICATION.md framing: the bypass is NOT at the `Canvas` matrix level (the Canvas has correct local coordinates within the `MobileLayerTransformer`'s rotated frame). The bypass is at the **shader-uniform level** — the painter has all the information it needs (the `MapCamera` snapshot it captured in `build()` carries `pixelOrigin`), but it forwards `(0,0)` to the shader instead of the real pan offset.

**Possibility #2** (`MobileLayerTransformer` applies translation only at the widget-Transform level so the Canvas inside doesn't see it) is **falsified by source**: `MobileLayerTransformer` doesn't translate at the widget level either; the rotation-only behaviour is by design and is consistent with how every other flutter_map layer works (each consumes `pixelOrigin` itself).

**Possibility #3** (camera updates don't propagate to `FogLayer.build()` between pan-driven repaint cycles) is **falsified by source**: `MapCamera.of(context)` is implemented via `MapInheritedModel.maybeCameraOf` (`map/inherited_model.dart` lines 36–38) which uses `InheritedModel.inheritFrom<MapInheritedModel>(context, aspect: _FlutterMapAspect.camera)`. The `updateShouldNotifyDependent` (lines 53–73) returns `true` whenever `data.camera != oldWidget.data.camera`, which fires on every pan-driven `MapInteractiveViewer.builder` rebuild (`map/widget.dart` lines 116–125). FogLayer IS rebuilt on every pan; the build does receive a fresh `MapCamera` snapshot; the painter's `cameraSnapshotMicros` advances. The reason rotation appears to propagate "differently" is that `MobileLayerTransformer.build()` itself reads `MapCamera.of(context).rotationRad` and applies `Transform.rotate(angle: rotationRad, ...)` — rotation is consumed at the widget-tree level by the transformer, while pan is consumed (or ignored) by each layer's per-paint projection. There is no separate setState path for rotation; both events flow through the same InheritedModel notification, and FogLayer's rotation handling works only because `MobileLayerTransformer` happens to read the camera in its own `build()` and apply the rotation to its child.

**Primary recommendation:** Fix `_FogPainter.paint()` to compute `offset` from `camera.pixelOrigin` and pass it through `shaderRenderer.render(...)`. The exact form: `offset: (camera.pixelOrigin.dx / size.width, camera.pixelOrigin.dy / size.height)` (UV-normalised since `noiseUv = fragUv + uOffset` and `fragUv` is normalised). Single-line fix at the painter; preserves the FOG-07 single-MapCamera-snapshot lock (the `camera` field already lives on the painter — passed by constructor in `build()`); does not touch identity `uSdfRect`; does not add `RepaintBoundary`; does not introduce multi-snapshot reads. Bundle this fix with full diagnostic instrumentation per the CONTEXT.md `decide-after-diagnostic` policy so a single sideload UAT walk either confirms the fix OR produces evidence enough to escalate to Phase 3.2.

### Pattern A: flutter_map Translation Propagation Convention (the architectural finding)

**What:** flutter_map 7.0.2 does NOT propagate translation through a Canvas matrix transform. There is no `MobileLayerTransformer` translation, no parent `Transform.translate`, no automatic `canvas.translate(camera.pixelOrigin)` injection. Instead, every blessed layer is responsible for consuming `MapCamera.pixelOrigin` (or one of its derived helpers — `getOffsetFromOrigin`, `latLngToScreenPoint`, `latLngToScreenOffset`) at its own paint or layout time.

**Verified by direct source-read:**

| Layer | How translation propagates | Source line(s) |
|-------|----------------------------|----------------|
| `TileLayer` (`tile.dart`) | `Positioned(left: tileCoord.x * scaledTileSize - currentPixelOrigin.x, top: ...)` per Tile widget | `lib/src/layer/tile_layer/tile.dart` lines 56–60 |
| `CircleLayer` (`circle_layer/painter.dart`) | `final center = camera.getOffsetFromOrigin(circle.point);` per circle per paint | lines 28, 53 |
| `PolygonLayer` (`polygon_layer/painter.dart`) | `final origin = (camera.project(camera.center) - camera.size / 2).toOffset();` then `getOffsetsXY(camera, origin: ...)` per polygon per paint | line 196 + 206 |
| `MobileLayerTransformer` (`shared/mobile_layer_transformer.dart`) | `OverflowBox(width: camera.size.x, ...)` + `Transform.rotate(angle: camera.rotationRad, ...)` — **no translation** | lines 19–24 |

**When to use (in our FogLayer):** populate `uOffset` from `camera.pixelOrigin` in `_FogPainter.paint()`. The `pixelOrigin` IS the world-pan vector from world-origin to the top-left of the camera's bounding rectangle — exactly the input the shader's noise-UV computation expects.

**Example (the fix):**
```dart
// Source: pixelOrigin definition — flutter_map 7.0.2 lib/src/map/camera/camera.dart line 77.
// pixelOrigin = project(center, zoom) - size / 2.0  (the world-space top-left of the camera's bounding box).
//
// uOffset semantics — assets/shaders/atmospheric_fog.frag line 65 + 258:
//   "World pan offset (in noise UV units). Lets the fog drift with the
//    MapLibre camera so static fixtures don't appear to move when the user
//    only pans."
//   noiseUv = fragUv + uOffset  →  fragUv is normalised [0,1]; so uOffset
//                                  must be in the same UV-normalised units.

@override
void paint(Canvas canvas, Size size) {
  if (sdfImage == null) return;
  final uTimeSeconds = wallClock.elapsedMicroseconds / _microsecondsPerSecond;
  final clipPath = computeFogClipPath(camera: camera, discs: discs);
  canvas.save();
  canvas.clipPath(clipPath);
  frameDeltaProbe.recordFogUniformPopulation(cameraSnapshotMicros);

  // FIX (Phase 3.1) — UV-normalised camera pan offset.
  // pixelOrigin is in flutter_map's projected-pixel space; size is in logical pixels;
  // the shader's noiseUv is fragUv + uOffset and fragUv is normalised. Dividing
  // pixelOrigin by size gives the right unit conversion. NB: pixelOrigin grows
  // unboundedly as the user pans — modulo'ing by 1.0 keeps numeric magnitudes
  // small without changing the noise pattern (FBM is periodic by construction).
  final pixOrigin = camera.pixelOrigin;
  final uOffsetX = (pixOrigin.x / size.width) % 1.0;
  final uOffsetY = (pixOrigin.y / size.height) % 1.0;

  shaderRenderer.render(
    shader: shader,
    resolution: size,
    timeSeconds: uTimeSeconds,
    offset: (uOffsetX, uOffsetY),                 // <-- THE FIX (was const (0.0, 0.0))
    baseAlpha: 1.0,
    sdfRect: const (0.0, 0.0, 1.0, 1.0),          // identity uSdfRect — UNCHANGED
    sdfImage: sdfImage!,
    mirkFogConstants: mirkFogConstants,
  );

  if (shader != null) {
    canvas.drawRect(Offset.zero & size, Paint()..shader = shader!);
  }
  canvas.restore();
}
```

The fix is a single derivation + one tuple constructor — three new lines of code in `paint()`. No new constructor argument, no `setState`, no new `MapCamera.of(context)` read (the `camera` field is already passed by constructor — single-snapshot lock preserved).

### Pattern B: Per-Paint JSONL Diagnostic Capture (FOG-10)

**What:** A new `FogTransformLogger` mirrors the existing `SdfRebuildLogger` shape: per-active-second emit one JSONL line via `Logger('infrastructure.mirk.fog_transform')` carrying a rollup of per-paint observations. The instrumented observation per paint:

| Field (suggested) | Source | Purpose |
|-------------------|--------|---------|
| `epochSecond` | `DateTime.now().millisecondsSinceEpoch ~/ 1000` | Wall-clock alignment with `frame_delta` + `sdf` rollups for post-walk grep-join |
| `sampleCount` | length of paint-observation buffer | sanity-check the rollup carried real data |
| `frameCounter` | monotonic counter incremented on every `paint()` call | density check |
| `canvasTransformTx`, `canvasTransformTy` | extracted from `canvas.getTransform()` Float64List elements 12 (tx) and 13 (ty) (column-major matrix4) | observed Canvas-local translation at `paint()` entry |
| `cameraPixelOriginX`, `cameraPixelOriginY` | `camera.pixelOrigin.x`, `camera.pixelOrigin.y` | the load-bearing field for the fix |
| `cameraCenterLat`, `cameraCenterLon` | `camera.center.latitude`, `camera.center.longitude` | sanity-correlate against GPS fix log |
| `uOffsetXAppliedSample`, `uOffsetYAppliedSample` | the actual `(uOffsetX, uOffsetY)` tuple passed to `shaderRenderer.render` | proves the fix landed in the binary |

**Rollup math:** for each numeric field, emit `min`, `median`, `max` over the per-second sample buffer (matches `SdfRebuildLogger` shape). `sampleCount` is the buffer length. Idle seconds (no paint calls) emit nothing — same convention as `SdfRebuildLogger`.

**Sample-rate tradeoff:** at 120 Hz on iPhone 17 Pro, capturing every paint produces ~120 samples/sec → one JSONL line per second is ≤ ~600 bytes (well under the 50 MB log smoke ceiling at ≤ 360 KB / 5-min walk; ≤ 4 MB / hour). **Per-paint capture is sustainable for 5+ min walks; no 1-in-N sampling required.** If the planner sees a need to bound memory for the buffer, cap at `kPocFrameDeltaBufferMaxSamples` (240) using the same FIFO-drop pattern as `FrameDeltaProbe._buffer`.

**Dual-clock discipline carries over:** `Stopwatch.elapsedMicroseconds` for any inter-paint timing, `DateTime.now().millisecondsSinceEpoch ~/ 1000` ONLY for the `epochSecond` rollup tag (REQUIRED for grep-correlation with `SdfRebuildLogger` and `FrameDeltaProbe`).

**Wire pattern:** the painter receives a `FogTransformLogger` reference by constructor (passed through `FogLayer` widget like `frameDeltaProbe`). At `paint()` entry: read `canvas.getTransform()`, read `camera.pixelOrigin` + `camera.center`, build the per-paint observation, push into the logger's buffer. The logger's `Timer.periodic(Duration(seconds: 1), ...)` rollups + emits.

**Instrumentation lifecycle:** kept permanently in production, always emitting at `Logger.info`. Per CONTEXT.md, NOT gated behind `--dart-define` flags — the log-timeline-alignment principle says future regressions surface immediately in the session log.

The verbosity tax — `didChangeDependencies` instead of `initState` (because `MapController.of(context)` requires the inherited model up the tree), event-type filtering, manual cancellation — is exactly why flutter_map's `MapInheritedModel` design uses InheritedModel aspects: dependents auto-rebuild when the camera aspect changes. Adding the manual stream subscription is at best redundant and at worst introduces a double-rebuild rate.

### Anti-Patterns to Avoid

- **Wrapping FogLayer's `CustomPaint` in `Transform.translate(camera.pixelOrigin, ...)`** — would double-translate the clip path (which already projects via `latLngToScreenPoint`) and visibly snap the fog by exactly `pixelOrigin` pixels. Reject.
- **Re-reading `MapCamera.of(context)` from inside `_FogPainter.paint()`** — re-introduces RESEARCH §Anti-Pattern 3 (multi-snapshot reads); painter must consume the snapshot captured in `build()`. The fix uses the painter's existing `camera` field, no re-read.
- **Touching identity `uSdfRect`** — RESEARCH §Anti-Pattern 1 (re-introduces BUG-014). The fix only modifies the `offset` slot 3..4; sdfRect slots 37..40 stay `(0,0,1,1)`.
- **Adding `RepaintBoundary` around `FogLayer`** — RESEARCH §Anti-Pattern 2 (would isolate from tile-layer repaint signal). Untouched.
- **Per-paint JSONL line emission (raw, not rolled-up)** — at 120 Hz × 5-min = 36k lines / walk, this would bloat the log file 60×. Rolled-up via `FogTransformLogger` (matches `FrameDeltaProbe` + `SdfRebuildLogger` discipline).
- **Capturing `BuildContext` in the painter for the diagnostic log** — same multi-snapshot trap as above. The painter logs the `pixelOrigin` it received via the `camera` constructor argument, never re-reads context.

## Common Pitfalls

### Pitfall A: Vacuous regression test (the FOG-04 lesson, made permanent)

**What goes wrong:** the natural shape of a regression test for "fog translates with map" is `find.descendant(of: FogLayer, matching: MobileLayerTransformer)` — which is exactly the FOG-04 structural test from Plan 03-05 that **passed during Plan 03-08 while production-fog was static**. The structural relationship is necessary but not sufficient.

**Why it happens:** widget-tree containment is the easy thing to assert with `find.*`. Behavioural transform-equality requires intercepting the painter's Canvas calls.

**How to avoid:** the new behavioural test must invoke `_FogPainter.paint(canvas, size)` against a `MockCanvas` that captures invocations, after a `MapController.move(LatLng(...), zoom)` programmatic drive, and assert that the `offset` argument to `shaderRenderer.render` is non-zero (and proportional to the camera's pan delta). Specifically: assert `(uOffsetX, uOffsetY) != (0, 0)` and `(uOffsetX, uOffsetY)` differs between two distinct camera positions. Vacuous-test guard: a `setUp` invariant asserts the test fails (RED) when the production code passes `offset: const (0.0, 0.0)` — i.e., the test must demonstrably fail against the current `main` HEAD (Plan 03-08 SHA 280dd04) before the fix is committed.

**Warning signs:** any regression test whose body is exhausted by a `find.*` or `expect(widget.foo, ...)` invariant — without ever invoking `paint()` against an instrumented Canvas — is structurally weaker than what FOG-04 already provides.

**Test seam:** see §Validation Architecture Test Shape (b) for the recommended construction.

### Pitfall B: Per-paint diagnostic budget at 120 Hz

**What goes wrong:** capturing `Canvas.getTransform()` (returns 16 doubles), reading 4 `camera.pixel*` doubles, allocating a per-paint observation object, and pushing into a buffer 120 times per second creates GC pressure that surfaces as frame jitter in the FrameDeltaProbe rollup.

**Why it happens:** `Float64List` allocation is a syscall on iOS; 120 allocations per second is observable in the probe's `max` field even if the median stays clean.

**How to avoid:** pre-allocate ONE long-lived `Float64List(16)` per painter instance (or one per `FogTransformLogger`); each paint call writes into the same buffer (the matrix elements that matter are indices 12 and 13 — `tx` and `ty`). Push only the 6 doubles that matter (tx, ty, pixelOrigin.x, pixelOrigin.y, center.lat, center.lon) into the buffer, NOT the full 16-element matrix. The buffer is `List<double>` of fixed capacity 240 (matches `FrameDeltaProbe._buffer` discipline), drop-oldest on overflow. Rollup math is `O(n log n)` over ≤ 240 elements once per second — ≈ 10 µs.

**Warning signs:** post-walk `frame_delta` rollup shows `maxMicros` spikes correlating in time with the fog-transform log's emission seconds.

**Test seam:** Phase 3.1 walk evidence — if the FrameDeltaProbe `maxMicros` trend changes vs the (uninstrumented) baseline, the diagnostic instrumentation itself is the contaminant. The post-walk grep correlates the two streams on `epochSecond`.

### Pitfall C: pixelOrigin grows unboundedly during long walks

**What goes wrong:** `MapCamera.pixelOrigin` is `project(center, zoom) - size / 2.0` — a world-projected pixel coordinate. At zoom 13–15 in central Melun, `project(center, zoom)` is on the order of `(2^15 * 256, 2^15 * 256) = (8.3 million, 8.3 million)`. Across a 5-min walk, deltas are small but the absolute values are large. Passing this directly to a `vec2` shader uniform is fine (`float` precision is plenty), but the `noiseUv = fragUv + uOffset` becomes `~8.3 million + small_uv`, and the FBM noise input loses precision because the `fract()`-based hash inputs are in the millions.

**Why it happens:** classic floating-point precision loss when adding a small delta to a large base.

**How to avoid:** modulo the offset by `1.0` before passing to the shader: `final uOffsetX = (pixOrigin.x / size.width) % 1.0;`. The FBM and curl-noise are spatially periodic on integer-modulo boundaries (the `fract()` of `noiseUv * uScale*` produces the same hash output for any `noiseUv + n` where `n` is integer), so modulo-1.0 preserves the visual pattern while keeping the float magnitudes tiny. **This is included in the Pattern A example fix above.**

**Warning signs:** post-walk visual inspection at zoom 17+ over a long pan distance shows fog noise becoming blocky / quantised — that's the precision-loss symptom.

**Test seam:** unit test on the modulo math: `assertion (pixOrigin.x / size.width) % 1.0 produces same FBM hash as raw value when fragUv = 0.5` — but this is a shader-equivalence assertion, not easily testable in pure Dart. Cheaper alternative: assert the helper `_uvOffsetFromPixelOrigin(pixOrigin, size)` returns values in `[0, 1)`.

### Pitfall D: `Canvas.getTransform()` at paint entry is the IDENTITY in MobileLayerTransformer at rotation=0

**What goes wrong:** the diagnostic instrumentation captures `canvas.getTransform()` expecting to see a translation matrix. At rotation=0 inside `MobileLayerTransformer` → `OverflowBox` → `CustomPaint`, the Canvas IS at identity (no translation, no rotation). The diagnostic line will show `(tx, ty) = (0, 0)` regardless of pan — making the log "look like" the fog isn't translating at the Canvas matrix level (because it isn't and shouldn't be, by flutter_map convention).

**Why it happens:** this is by design — flutter_map propagates translation per-layer-by-convention, NOT via Canvas matrix. See Pattern A.

**How to avoid:** the diagnostic log must capture BOTH `canvas.getTransform()` (proves Canvas matrix is identity) AND `camera.pixelOrigin` (proves the camera DID receive the pan info). The diff is the smoking gun: Canvas-tx=0 but pixelOrigin.x grows during pan ⇒ the painter has the data, just isn't using it. That diff IS the diagnostic deliverable. Without capturing both, the log is ambiguous.

**Warning signs:** post-walk log shows `canvasTransformTx` and `canvasTransformTy` flat at 0 while `cameraPixelOriginX/Y` track the user's panning. That's expected — the data is the diff between them, not the absolute Canvas matrix value.

**Test seam:** the FOG-09 regression test's MockCanvas asserts `getTransform()` returns identity-equivalent (within epsilon — see Pitfall E) when `_FogPainter.paint()` is called. The fix proves itself by passing non-zero `offset` to `shaderRenderer.render`, NOT by changing the Canvas matrix.

### Pitfall E: Float-equality on `Canvas.getTransform()` matrix elements

**What goes wrong:** `Float64List` is comparable element-wise but accumulated transforms (e.g., due to internal `RenderObject` Offset compositing) introduce sub-pixel rounding. A test asserting `tx == 0.0` fails on a Canvas that's actually been translated by `(0, 0.0000001)`.

**Why it happens:** `Canvas` matrix accumulates `save`/`translate`/`restore` calls; sub-pixel rounding is normal.

**How to avoid:** all matrix-element comparisons in the regression test use `closeTo(expected, kPocCanvasTransformEpsilon)` where `kPocCanvasTransformEpsilon = 1e-6` (or whatever the planner picks based on the camera-driven translation per pan-pixel — likely 1e-3 logical pixels at zoom 13). Include an explicit constant in `lib/config/constants.dart` so the choice is documented and testable.

**Warning signs:** flaky regression test that passes locally and fails on CI on macOS / Linux runner.

**Test seam:** see §Code Examples Example 4.

### Example 1: `FogTransformLogger` (mirror of `SdfRebuildLogger`)

```dart
// Source: pattern derived from lib/infrastructure/mirk/sdf_rebuild_logger.dart
// (Plan 03-03) — same 1-Hz cadence, same JSONL field shape, same
// idle-second-skip behaviour. Fields specialised for fog transform
// diagnostics per CONTEXT.md.
class FogTransformLogger {
  FogTransformLogger({Duration? rollupInterval})
      : _rollupInterval = rollupInterval ?? const Duration(seconds: kPocFogTransformLogRollupSeconds);

  static final Logger _log = Logger('infrastructure.mirk.fog_transform');

  final Duration _rollupInterval;
  final List<_FogTransformSample> _buffer = <_FogTransformSample>[];
  Timer? _timer;
  int _frameCounter = 0;

  void start() {
    _timer ??= Timer.periodic(_rollupInterval, (_) => _emitRollup());
  }

  void stop() {
    _timer?.cancel();
    _timer = null;
    if (_buffer.isNotEmpty) _emitRollup();
  }

  /// Record one paint observation. Called from `_FogPainter.paint()` AFTER
  /// the camera-pixelOrigin has been read but BEFORE the shader uniforms
  /// are populated.
  void recordPaint({
    required Float64List canvasTransform,    // 16-element column-major
    required Point<double> cameraPixelOrigin,
    required LatLng cameraCenter,
    required (double, double) appliedUOffset,
  }) {
    _frameCounter++;
    _buffer.add(_FogTransformSample(
      frameCounter: _frameCounter,
      canvasTx: canvasTransform[12],
      canvasTy: canvasTransform[13],
      pixelOriginX: cameraPixelOrigin.x,
      pixelOriginY: cameraPixelOrigin.y,
      centerLat: cameraCenter.latitude,
      centerLon: cameraCenter.longitude,
      uOffsetX: appliedUOffset.$1,
      uOffsetY: appliedUOffset.$2,
    ));
    while (_buffer.length > kPocFogTransformBufferMaxSamples) {
      _buffer.removeAt(0);
    }
  }

  void _emitRollup() {
    if (_buffer.isEmpty) return;
    // ... compute min/median/max for the 8 fields ...
    final epochSecond = DateTime.now().millisecondsSinceEpoch ~/ 1000;
    final line = json.encode(<String, Object>{
      'epochSecond': epochSecond,
      'sampleCount': _buffer.length,
      // ... fields ...
    });
    _log.info(line);
    _buffer.clear();
  }
}

  // immutable fields ...
}
```

### Example 2: `_FogPainter.paint()` after the fix + instrumentation wire

```dart
@override
void paint(Canvas canvas, Size size) {
  if (sdfImage == null) return;

  final uTimeSeconds = wallClock.elapsedMicroseconds / _microsecondsPerSecond;
  final clipPath = computeFogClipPath(camera: camera, discs: discs);
  canvas.save();
  canvas.clipPath(clipPath);

  frameDeltaProbe.recordFogUniformPopulation(cameraSnapshotMicros);

  // FIX: derive UV-normalised pan offset from the camera's pixelOrigin.
  // pixelOrigin is in projected-pixel space; size.width/height are in logical pixels;
  // the shader's noiseUv = fragUv + uOffset and fragUv is normalised. Modulo-1.0
  // keeps float magnitudes tiny (FBM is integer-periodic so visually identical).
  final pixOrigin = camera.pixelOrigin;
  final uOffsetX = (pixOrigin.x / size.width) % 1.0;
  final uOffsetY = (pixOrigin.y / size.height) % 1.0;
  final appliedOffset = (uOffsetX, uOffsetY);

  // FOG-10 diagnostic capture — single Float64List allocation per paint
  // (16 doubles), pre-allocated on the painter for GC budget.
  final transform = canvas.getTransform();   // returns the existing _diagnosticBuffer
  fogTransformLogger.recordPaint(
    canvasTransform: transform,
    cameraPixelOrigin: pixOrigin,
    cameraCenter: camera.center,
    appliedUOffset: appliedOffset,
  );

  shaderRenderer.render(
    shader: shader,
    resolution: size,
    timeSeconds: uTimeSeconds,
    offset: appliedOffset,                    // <-- THE FIX (was const (0.0, 0.0))
    baseAlpha: 1.0,
    sdfRect: const (0.0, 0.0, 1.0, 1.0),      // identity — UNCHANGED
    sdfImage: sdfImage!,
    mirkFogConstants: mirkFogConstants,
  );

  if (shader != null) {
    canvas.drawRect(Offset.zero & size, Paint()..shader = shader!);
  }
  canvas.restore();
}
```

## Open Questions

1. **Whether `pixelOrigin` modulo-1.0 in UV space is visually identical to raw pixelOrigin during slow zoom transitions**
   - What we know: FBM and curl-noise are spatially periodic on integer-modulo boundaries (the `fract()`-based hash inputs produce identical outputs for `noiseUv + n` where `n` is integer). At constant zoom this is exact. During zoom transitions the period changes and the modulo-1.0 transition between frames could theoretically introduce a one-frame discontinuity.
   - What's unclear: whether the discontinuity is visually perceptible at the POC's zoom range (13–17) and pan velocities (~human walking speed = pixels/frame is small).
   - Recommendation: planner picks the modulo'd form by default per Pitfall C; the diagnostic log's `uOffsetXAppliedSample` field captures the actual value emitted per frame so the post-walk evidence can demonstrate (or refute) any visual discontinuity. If discontinuity surfaces during the walk, the fallback is no-modulo with the precision-loss tradeoff documented and the planner picks an alternative (e.g., subtract a per-session offset captured at startup).

2. **Whether the regression test should drive `MapController.move` once-per-frame or burst-pan**
   - What we know: Plan 03-08 walk exhibited the static-fog symptom on continuous pan gestures, not on a single discrete move. A single discrete `MapController.move()` call DOES rebuild the InheritedModel and therefore the FogLayer; the test asserts the uniform changed across that one move event.
   - What's unclear: whether continuous-pan-during-paint introduces a different failure mode (e.g., a paint cycle that sees a stale `cameraSnapshotMicros` because the gesture system is faster than the build phase) — this would be possibility #3-flavoured but limited to the gesture-system's drag handler.
   - Recommendation: Test Shape (b) asserts the discrete move case — the most-tractable test that catches the documented failure mode. If walk evidence shows a continuous-pan-specific symptom not covered by the discrete test, the planner adds a follow-up test (out of Phase 3.1 scope unless the walk surfaces it).

3. **Whether `MapScreenServices` should expose the `MapController` instance as a thin DI seam**
   - What we know: the regression test needs a `MapController` to drive programmatic moves. Production currently does NOT thread a `MapController` through `MapScreenServices` — `FlutterMap` creates one internally.
   - What's unclear: whether the test can grab the internal controller via `MapController.of(context)` from a test seam, or whether a constructor-injected controller is cleaner.
   - Recommendation: planner picks. Cheapest path: the regression test instantiates its own `MapController()` and passes it to the test's `FlutterMap`, the production app stays unchanged. If the planner sees value in unifying the test + production wiring (e.g., for future `MapView` abstraction work — out of POC scope), the `MapScreenServices.mapController` field is a clean DI extension.
