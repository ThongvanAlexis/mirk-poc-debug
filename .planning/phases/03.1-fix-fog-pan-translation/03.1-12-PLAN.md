<objective>
Close the Q5 zoom-scramble failure mode that Walk #4 (Plan 03.1-11) surfaced at the architectural noise-anchor level. The Plan 03.1-10 layered fix bundle (UX-02 + FOG-17 + FOG-17a) walk-time-validated all four pan-axis sub-claims (Q1+Q2+Q3+Q4) but the FOG-17 formulation `noiseUv = worldPx / kNoiseTilePx` anchors noise to **Web-Mercator world-pixel space** with `uPixelOrigin` doubling per zoom level → fragments under any given screen position sample completely different noise positions per zoom step (numbers slide fast); combined with `kNoiseTilePx = 384.0` raw-pixel const → cells pinned in screen-space rather than scaling with the world (numbers stay same size).

**FOG-18 (D-1, primary fix axis — DEVELOPER-ENDORSED)**: world-meter anchor. Replace the FOG-17 pixel-space anchor formulation with a meter-space anchor formulation:

```glsl
// Pre-Plan-03.1-12 (FOG-17 — pixel-space anchor):
vec2 worldPx = fragUv * uResolution + uPixelOrigin;
vec2 noiseUv = worldPx / kNoiseTilePx;     // kNoiseTilePx = 384.0 raw px

// Post-Plan-03.1-12 (FOG-18 — meter-space anchor):
vec2 worldMeters = (fragUv * uResolution + uPixelOrigin) * uMetersPerPixel;
vec2 noiseUv = worldMeters / kNoiseTilePxMeters;     // kNoiseTilePxMeters = 1024.0 m
```

Each fragment now samples noise at its own world-meter position. Zoom-out/zoom-in scales `uMetersPerPixel` such that `worldMeters` evaluates to the SAME value at the same geographic point regardless of zoom — the noise pattern is anchored to ground meters and zoom-invariant in geographic terms. Cells appear larger when zoomed in (more raw px per meter ground) and smaller when zoomed out, exactly matching the developer's expected behaviour from Walk #4 Q5 verbatim *"I should see the number getting bigger"*. NEW v1 requirement **FOG-18** captures this acceptance.

**Implementation specifics (developer-endorsed via Walk #4 close-out — see `03.1-FALSIFICATION-4.md` Sub-section D hypothesis triage table):**

1. NEW uniform `uMetersPerPixel` at slot 41 (`FogShaderUniforms.totalFloatSlots` advances 41 → 42; under shader budget cap of 64 — Flutter Impeller's actual hard limit is well above the conservative POC scoping value).
2. Shader change in BOTH `assets/shaders/atmospheric_fog.frag` AND `assets/shaders/atmospheric_fog_debug_spiral.frag` (symmetric — debug-spiral cell-numbering reflects the meter-space coordinate system; cells are now physical squares of ground regardless of zoom).
3. <noise_function_inspection_under_meter_space>
The production shader's noise stack is unchanged from Plan 03.1-10 — `hash3 → noise3 → fbm3` with rotation matrices between octaves. What changes is the input space: post-FOG-18, `noiseUv` is computed in meter-space rather than pixel-space.

**Continuity invariant in meter space:** at a fixed geographic point (fixed lat/lng), `(fragUv * uResolution + uPixelOrigin) * uMetersPerPixel` evaluates to the SAME value regardless of zoom — this is the FOG-18 acceptance property. Mechanistically: as zoom doubles, `uPixelOrigin` doubles AND `(fragUv * uResolution)` stays the same (the viewport-pixel position of the fragment is screen-space, zoom-invariant) AND `uMetersPerPixel` halves. So `((fragUv * uResolution + uPixelOrigin) * uMetersPerPixel)`'s zoom-derivative is `((0 + 2 * uPixelOrigin) * 0.5 * uMetersPerPixel) = (uPixelOrigin * uMetersPerPixel)` — wait, that's not quite zoom-invariant; let me re-derive carefully.

Web-Mercator standard: at zoom z, the world is `256 * 2^z` raw pixels wide. The camera's `pixelOrigin` at a fixed geographic point (lat0, lon0) is `pixelOrigin(z) = pixelFromLatLng(lat0, lon0, z)` which scales as `2^z`. Specifically `pixelFromLatLng(lat, lon, z) = ((lon + 180) / 360) * 256 * 2^z`. So `pixelOrigin.x` doubles per zoom step at the same geographic position.

Now `fragUv` is in [0, 1] (viewport-relative); `fragUv * uResolution` is in raw screen pixels. The viewport's screen size doesn't change with zoom (still ~430 raw px wide on iPhone 17 Pro). For a fragment at a fixed SCREEN position, `fragUv * uResolution` is constant across zoom.

But for a fragment at a fixed GEOGRAPHIC position, the screen position changes with zoom (the geographic point projects to different screen pixels at different zoom levels). At the SAME geographic point, `(fragUv * uResolution + uPixelOrigin)` evaluates to the WORLD-PIXEL coordinate of the geographic point at zoom z, which is `pixelFromLatLng(lat, lon, z) = ((lon + 180) / 360) * 256 * 2^z` — scales as `2^z`.

Multiplying by `uMetersPerPixel(z) = kWebMercatorMetersPerPxAtEquatorZ0 * cos(lat) / 2^z` (scales as `2^-z`), we get `worldMeters = pixelFromLatLng(lat, lon, z) * metersPerPixel(z) = ((lon + 180) / 360) * 256 * 2^z * 156543.03 * cos(lat) / 2^z = ((lon + 180) / 360) * 256 * 156543.03 * cos(lat)` — INDEPENDENT of zoom. **QED**: at a fixed geographic point, `worldMeters` is zoom-invariant. The noise sample at that geographic point is therefore zoom-invariant: FOG-18 acceptance property is mechanically guaranteed by the formula.

**Caveat — FBM rotation matrices:** the shader's FBM applies rotation matrices between octaves (`p = p * 2.03 + vec3(13.7, 7.3, 5.1)` between octaves 1→2, etc.). These rotations are linear and zoom-invariant — they operate on `noiseUv` AFTER the meter-space anchor lands. So the FBM output at a fixed geographic point IS zoom-invariant under FOG-18.

**Integer-wrap continuity in meter space:** FOG-17a's pixel-space integer wrap (every 1536 raw px) translates to a meter-space wrap of `1536 * uMetersPerPixel` meters. At zoom 15 lat 48.5°, that's `1536 * 3.15 ≈ 4838 m` of pan per wrap. The wrap shifts `noiseUv` by `(1536 * 3.15) / 1024 ≈ 4.72` cells — NOT integer-multiple. The base-octave `noise3` is periodic with period 1.0 in noise-grid units, so a non-integer shift produces a discontinuity in the base octave (not preserved across the wrap). The FBM rotation matrices preserve geometric continuity but not exact-period continuity at non-integer shifts.

**Resolution:** at zoom 15 walk velocity 12 raw px/s, the wrap fires every ~128 sec — same cadence as pixel-space FOG-17a. The empirical Walk #1+2+3 evidence showed ~3-sec stepping is perceptible but ~128-sec stepping is below the perceptual threshold (Walk #3b empirical anchor: median 39 raw px between markers at 16-65 raw px wrap period). A non-integer-multiple shift at one moment every ~128 sec is therefore expected to remain sub-perceptible. **Plan 03.1-13+ contingency:** if Walk #5 surfaces residual wrap-stepping at the 128-sec cadence, two paths are available: (a) pivot to a periodic-noise function (Worley/explicit-period Perlin) whose lattice IS preserved under any shift — eliminates the wrap problem entirely; (b) make `kPocFogIntegerWrapPeriodPx` zoom-dependent so the meter-space wrap is always an integer-multiple of `kNoiseTilePxMeters` at the active zoom — adds Dart-side complexity but stays within the existing decomposition framework. This is documented in the constant docstring.

**Documentation:** the meter-space anchor + uniform layout change + FOG-17a-meter-space chain + Walk #5 contingency are documented in (a) `kPocFogNoiseTilePxMeters` constant docstring; (b) the production shader source's `noiseUv` computation block comment; (c) `_FogPainter.paint()` metersPerPixel computation comment block; (d) the FogShaderUniforms docstring slot-layout table. Future regressions (POC OR MirkFall after port-back) read the rationale at the call site without needing to re-derive it.
