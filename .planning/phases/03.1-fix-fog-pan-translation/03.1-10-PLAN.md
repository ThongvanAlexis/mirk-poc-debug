**FOG-17 (D-1, primary fix axis)**: world-coordinate noise sampling. Replace the Plan 03.1-07 Branch B-3 partial-fix formulation `noiseUv = fragUv + fract(uPixelOrigin / tilePeriodPixels)` (which moved the wrap period from ~390 px to ~16-65 px but kept the per-event wrap MAGNITUDE the same — Walk #3b empirical anchor: 8 markers, median 39 raw px between markers, 2.4 wrap events per perceived step) with `worldPx = fragUv * uResolution + uPixelOrigin; noiseUv = worldPx / kNoiseTilePx`. This is the developer's correctly-intuited iteration path from Walk #3 Q1 verbatim *"if I pan right forever the shader should not be moved to be where I'm going, I should see a new area of the shader"*.

**FOG-17a (D-1a precision pairing)**: CPU-side integer/fractional decomposition. Pure FOG-17 (`worldPx = fragUv * uResolution + uPixelOrigin`) exposes fp32 precision degradation at high zoom — Walk #2 captured pixelOriginX up to 4.26M; fp32 ULP at that magnitude is ≈0.5 raw px, which is catastrophic for sub-noise-cell sampling. The decomposition splits `camera.pixelOrigin` into integer + fractional Dart-side and forwards a bounded `(intPxX % kPocFogIntegerWrapPeriodPx + fracPxX, intPxY % kPocFogIntegerWrapPeriodPx + fracPxY)` tuple to the shader so the shader input stays under ~1537 raw px regardless of zoom. The integer wrap period is an integer multiple of the noise tile period so the wrap event produces an identical noise pattern (the FBM samples the same hash3 grid corners on both sides of the wrap).

**UX-02 + FOG-16 path (a)**: disable map rotation. `FlutterMap.interactionOptions` changes from `InteractionOptions(flags: InteractiveFlag.all)` to `InteractionOptions(flags: InteractiveFlag.all & ~InteractiveFlag.rotate)`. Walk #3 Q2 surfaced a NEW rotation-correlated fog mis-coverage failure mode: Plan 03.1-08's `canvas.translate(-canvasOffset)` compensates ONLY translation (matrix[12], matrix[13]); the rotation component (matrix[0,1,4,5]) is untouched, so MobileLayerTransformer rotation during pinch-zoom-rotate causes the fog rect to rotate with the canvas leaving wedges of un-fogged map at viewport corners. Disabling rotation structurally prevents this — `MapCamera.rotation` stays at 0.0 always, so the canvas transform never accumulates rotation matrix elements. Developer-endorsed POC scope reduction per Walk #3 verbatim *"I think we should disable rotation for now, only pan and zoom, north is up. simplier"*. FOG-16 path (b) full canvas-inverse-transform stays deferred to hypothetical post-POC.

1. `hash3(p) = fract(p * 0.1031); p += dot(p, p.yxz + 19.19); return fract((p.x + p.y) * p.z);` — deterministic per `vec3` input; NOT integer-translation-equivariant in general.
2. `noise3(p)` = trilinear interp of hash3 at the 8 unit-cube corners around `p`. Continuous in `p`.
3. `fbm3(p)` = 3-octave sum of `noise3` with rotation matrices between octaves: `p = p * 2.03 + vec3(13.7, 7.3, 5.1); a *= 0.5;` (octave 1→2), `p = p * 2.05 + vec3(-11.1, 17.9, 3.3); a *= 0.5;` (octave 2→3).
4. The shader's main path: `noiseUv` (post-fix world-coordinate formulation) → multiplied by `uScaleFar/Mid/Near` per octave → fed into `fbm3(vec3(warped, uTime * driftZ))` → density.

**Continuity invariant under FOG-17a integer wrap:** at the wrap boundary, the Dart-side decomposition shifts the shader's `uPixelOrigin` input by exactly `kPocFogIntegerWrapPeriodPx` (1536) raw pixels. The shader's `noiseUv = (fragUv * uResolution + uPixelOrigin) / kNoiseTilePx` therefore shifts by exactly `1536 / 384 = 4` noise grid units. Per-octave: `warped = noiseUv * uScale*` shifts by `4 * uScale*` per octave — `4 * 2.9 = 11.6`, `4 * 5.1 = 20.4`, `4 * 10.5 = 42.0`. `noise3` is periodic with period 1.0 in each of its three input axes (the hash3 lattice has period 1 by construction since `floor(p)` quantises to integer corners). `11.6`, `20.4`, `42.0` are NOT integer-period offsets, so the FBM output IS NOT preserved under the integer-wrap shift in general.

**Resolution:** the wrap event produces a discontinuity in the FBM output at the wrap boundary — but this discontinuity IS already gated by the integer wrap period (~1536 raw pixels of camera pan) which at typical pan velocities (10-50 raw px/sec) corresponds to ~30-150 seconds of continuous one-direction pan. The user perceiving this discontinuity at all requires sustained pan in one direction crossing 1536 raw px. Walk #3b's empirical anchor: ~12 raw px/s mean pan velocity, so an integer wrap fires every ~128 seconds of continuous pan. By contrast, the pre-Plan-03.1-10 stepping fired every ~3 seconds (every ~16-65 raw px). FOG-17a moves the wrap from "every few seconds" to "every two minutes or more" — a perceptual reduction by a factor of ~40, sufficient for the Walk #4 verdict gate.

**If Walk #4 surfaces residual integer-wrap stepping:** Plan 03.1-12+ would adopt a more sophisticated decomposition strategy (e.g., add a second wrap-period uniform that aligns with the FBM rotation matrices, OR pivot to a different noise function entirely — e.g., a periodic Worley/Perlin noise with explicit period-1 lattice). This is documented as the iteration path; not part of Plan 03.1-10's scope.

**Alternative considered + rejected:** sample noise at `noiseUv % kHashLatticePeriod` (modulo within the shader). This re-introduces a per-fragment fract-style discontinuity at every hash-lattice boundary (period 1.0 in noise grid units = `kPocFogNoiseTilePx` = 384 raw pixels) — strictly WORSE than the FOG-17a strategy (wrap events fire every 384 raw px instead of every 1536 raw px).

**Documentation:** the integer-wrap-period choice + the noise-function inspection rationale + the Walk #4 contingency are documented in the `kPocFogIntegerWrapPeriodPx` constant docstring + a comment block in `_FogPainter.paint()` above the decomposition lines. Future regressions (POC OR MirkFall after port-back) read the rationale at the call site without needing to re-derive it.
