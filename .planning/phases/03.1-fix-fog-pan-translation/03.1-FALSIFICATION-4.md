### NEW Q1b failure mode — zoom-gesture-correlated very-fast hard-step ("seed-of-procedural-pattern-fast")

Developer's verbatim: *"zoom in/out : very fast hard step looking like changing the seed of a procedural generated image very fast"*.

This is a **NEW failure mode surfaced for the first time in Walk #4** — distinct from Q1 (pure-pan stepping). The fog_transform stream confirms the mechanism candidate:

Rollups 20-24 (zoom-out gesture, epochSec 1777891847..1777891851, ~5 sec): pixelOriginX sweeps **4.10M → 3.42M → 0.93M → 0.84M → 0.81M** (4 distinct order-of-magnitude jumps in 5 seconds). uOffsetXMax/YMax fluctuate across the full bounded range (1499-1532). At each zoom-step boundary, the world-pixel-per-screen-pixel ratio changes (since `uResolution` stays constant but `uPixelOrigin` jumps by ~3M), so `worldPx = fragUv * uResolution + uPixelOrigin` jumps non-continuously across the zoom-step boundary.

**Mechanism (M-Q1b):** the world-coordinate noise sampling formulation gives **noise-value continuity in pure-pan space** (camera moves; uPixelOrigin moves smoothly; worldPx moves smoothly; noise samples move smoothly). It does NOT give noise-value continuity in **zoom-gesture space** (camera zoom changes; uPixelOrigin jumps discontinuously by O(1M) raw px between zoom steps; worldPx jumps; noise samples re-randomise). Per Walk #2's empirical data + Walk #4's rollup-20-24 trajectory, flutter_map's zoom is **NOT smoothly interpolated** at the pixel-origin level — it snaps between zoom integer levels (or discrete sub-levels), producing a jump in pixelOrigin at each snap.

**Pre-Plan-03.1-10 the symptom was masked** because (a) `fract(uPixelOrigin / *)` collapsed the wrap-induced discontinuity into a smaller visible jump, AND (b) the rotation-correlated Walk #3 failure mode dominated user attention. Walk #4 surfaces Q1b because the layered fix bundle removed the dominant failure modes, leaving the previously-second-order zoom-gesture mode visible.

**Plan 03.1-12+ candidate fix axes for Q1b:**

- **(C-Q1b-1) seed-stable noise across zoom transitions** — pin a per-paint seed that decouples noise-sample position from pixelOrigin transitions, so the noise pattern at world (lat, lon) stays the same across zoom levels (just sampled at different scale). Mathematically: replace `noiseUv = worldPx / kNoiseTilePx` with `noiseUv = worldLatLon * kNoiseFrequencyPerLatLon` where `worldLatLon = projectFromPixelOrigin(uPixelOrigin)`. Trades shader-side complexity (project pixelOrigin → lat/lon every fragment) for zoom-invariance of the noise pattern.
- **(C-Q1b-2) interpolate uPixelOrigin across zoom-snap boundaries** at the Dart side — when MapEventMove fires a zoom delta > threshold, animate uPixelOrigin smoothly between the pre-zoom and post-zoom values over N frames. Trades responsiveness (zoom feels laggy) for noise-sample continuity.
- **(C-Q1b-3) accept Q1b as POC-acceptable** — zoom is a discrete user-initiated transition; the noise pattern flashing during the gesture is arguably less objectionable than a frozen fog. This is the lowest-effort path; defer fix to MirkFall port-back where the production fog UX may have other zoom-time visual treatments (fade, slow zoom curve, etc.) that mask the seed-flash.

### Q1 mechanism correction — SNAP, not seam

The developer's verbatim characterization of Q1 at MAX zoom (added post-walk-4): *"i don't think it's a seam, it looks like the mirk is translated HARD AND FAST in 1 millisecond when panning at max zoom, also to be clear it does not happen all the time like while zooming, you can pan a little bit then at some point SNAP, then it's good again"*.

This **invalidates M1 / M2 / M3** as originally framed (those candidates assumed a continuous-but-imperfect noise-tile-boundary seam). The correct mechanism is:

**(M1') FOG-17a integer-wrap-modulo firing visibly at MAX zoom.** Plan 03.1-10 set `kPocFogIntegerWrapPeriodPx = 1536.0` raw-px (= `4 × kPocFogNoiseTilePx (=384.0)`) under the assumption that the noise function is exactly periodic on 384 — so a wrap at any multiple of 384 (including 1536) is invisible by construction. The observation that wraps are visible **at MAX zoom only** falsifies that assumption: either (a) the noise function isn't truly periodic on 384, or (b) fp32 drift at MAX-zoom worldPx ≈ 4.26M makes the modulo land on a non-aligned offset, or (c) both.

The temporal pattern fits: at default zoom 1536 raw-px ≈ many seconds of pan → wraps rare → invisible (also the noise's mismatch from periodicity is below visual threshold at lower magnification). At MAX zoom 1536 raw-px ≈ a few cm of finger drag → wraps frequent → visible SNAPs every few seconds during sustained pan, intermittently during typical gesture cadence. Walk #4 marker timing matches: 2 markers in 2.36 sec at ~47 raw-px/s = ~110 raw-px between markers, well under the 1536-raw-px wrap period — so each marker is one wrap event, not multiple.

The debug-spiral's silence on Q1 during pan at MAX zoom is the positive control: the digit-atlas pattern **is** truly periodic on 384 (it's a deterministic cell grid with explicit period), so the wrap at 1536 lands on an aligned cell boundary and is genuinely invisible. The noise function does NOT have that property in practice.

### Plan 03.1-12+ Dart-only fix axes (corrected)

Both shaders are LOCKED. Fix candidates are at the call site (`_FogPainter.paint()` in `lib/presentation/widgets/fog_layer.dart`) + constants (`lib/config/constants.dart`) + map-screen options (`lib/presentation/screens/map_screen.dart`).

**(C1') Q1 SNAP axis — relax or eliminate the FOG-17a integer-wrap-modulo.** fp32 has 24 bits of exact-integer mantissa = 16.7M raw-px; Walk #4 logs show pxOrigin ≤ 4.26M — well within float32's exact-integer range. The integer-wrap was added "for safety" to bound shader-side precision, but Walk #4 demonstrates the precision penalty at 4.26M is not what's producing user-visible artefacts — the WRAP itself is.   - **(C1'-recalibrate)** widen the wrap period to a value empirically calibrated against the noise function's actual period (e.g. probe via offline shader inspection or empirical walk-time A/B). Pure constants change in `lib/config/constants.dart`.
